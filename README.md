# PRS Express TypeScript API

This project is a Node.js backend built with Express, TypeScript, MongoDB, and Mongoose.
It supports:

- uploading a PRS report JSON file
- storing the report in MongoDB
- merging new phenotypes into the same report when the same `sample.id` is uploaded again
- fetching report summaries, categories, and phenotypes
- viewing API documentation through Swagger

## Folder Purpose

- `src/config`: environment and database setup
- `src/controllers`: handles incoming requests and sends responses
- `src/docs`: OpenAPI / Swagger documentation
- `src/helpers`: small reusable helper functions for shaping data
- `src/middlewares`: shared request/response middleware
- `src/models`: Mongoose models and schemas
- `src/routes`: API route definitions
- `src/services`: business logic and data access logic
- `src/types`: shared TypeScript types
- `src/utils`: common reusable utilities
- `dist`: compiled JavaScript output generated by the build
- `node_modules`: installed npm packages

## Root Files

### `.env`
Stores real local environment variables used when the app runs.

Current use:

- app port
- environment mode
- API prefix
- MongoDB connection string
- optional OAuth bearer token verification settings

### `.env.example`
Sample environment file for reference.
Use this as a template when setting up the project on another machine.

OAuth is disabled by default with `AUTH_ENABLED=false`, so API requests do not need a token right now.
Later, set `AUTH_ENABLED=true` and configure `OAUTH_INTROSPECTION_URL` with the OAuth client settings to require `Authorization: Bearer <token>` for API routes.

### `.gitignore`
Tells Git which files and folders should not be committed.
For example:

- `node_modules`
- `dist`
- `.env`

### `package.json`
Main npm configuration file.
It defines:

- project name and metadata
- dependencies and dev dependencies
- npm scripts such as `dev`, `build`, `start`, and `type-check`

### `package-lock.json`
Locks the exact installed package versions so installs stay consistent across machines.

### `tsconfig.json`
TypeScript compiler configuration.
It controls how TypeScript is compiled into JavaScript.

### `README.md`
This documentation file.
It explains the project structure and the purpose of each file.

## Generated Folders

### `dist/`
Contains compiled JavaScript generated by `npm run build`.
You usually do not edit files here manually.

### `node_modules/`
Contains all installed npm packages.
This is also generated and should not be edited manually.

## Source Files

### `src/app.ts`
Creates and configures the Express application.

Responsibilities:

- enables security middleware with `helmet`
- enables `cors`
- enables JSON and URL-encoded body parsing
- enables request logging with `morgan`
- exposes `/docs` and `/docs.json`
- applies optional OAuth verification before API routes
- mounts all API routes under the configured API prefix
- attaches not-found and error middleware

### `src/server.ts`
Application entry point.

Responsibilities:

- connects to MongoDB
- starts the HTTP server
- listens on the configured port
- handles graceful shutdown for `SIGINT` and `SIGTERM`

## `src/config`

### `src/config/database.ts`
Contains MongoDB connection logic using Mongoose.
This is where the application connects to the database defined in `.env`.

### `src/config/env.ts`
Loads and validates environment variables using `dotenv` and `zod`.
This keeps runtime configuration safe and typed.

## `src/controllers`

### `src/controllers/user.controller.ts`
Controller for user-related endpoints.

Responsibilities:

- create a user
- get all users
- get a user by id

This file only handles request/response flow and delegates business logic to the service.

### `src/controllers/report.controller.ts`
Controller for report-related endpoints.

Responsibilities:

- upload a report JSON file
- fetch a report by report id
- fetch report categories
- fetch a single phenotype by phenotype id
- fetch all phenotypes inside a specific category

This file keeps controllers thin and delegates logic to `report.service.ts`.

## `src/docs`

### `src/docs/openapi.ts`
Defines the OpenAPI / Swagger specification for the project.

Responsibilities:

- documents available API endpoints
- powers `/docs.json`
- powers the Swagger UI at `/docs`

When a new API is added, this file should be updated too.

## `src/helpers`

### `src/helpers/user.helper.ts`
Helper functions related to user data.

Responsibilities:

- build a full display name
- shape the public user response object

### `src/helpers/report.helper.ts`
Helper functions related to report data.

Responsibilities:

- parse uploaded JSON report files
- shape report response objects
- extract category names from phenotypes
- build category summary responses
- build phenotype detail responses
- build category-specific phenotype list responses

This file is mainly for transforming raw report data into API-friendly output.

## `src/middlewares`

### `src/middlewares/error.middleware.ts`
Centralized error handler.

Responsibilities:

- catches application errors
- handles Zod validation errors
- handles Mongoose errors
- handles Multer upload errors
- sends a consistent error response format

### `src/middlewares/notFound.middleware.ts`
Handles requests to routes that do not exist.
Returns a clean 404 error using the shared `ApiError` class.

### `src/middlewares/oauth.middleware.ts`
Optional OAuth bearer token middleware.
It is disabled by default and only verifies tokens when `AUTH_ENABLED=true`.

### `src/middlewares/uploadJson.middleware.ts`
Handles JSON file upload configuration using `multer`.

Responsibilities:

- stores uploaded files in memory
- limits file size
- allows only `.json` files
- expects the upload field name to be `file`

### `src/middlewares/validateRequest.middleware.ts`
Reusable request validation middleware using Zod schemas.
It validates `body`, `params`, and `query` before the controller runs.

## `src/models`

### `src/models/user.model.ts`
Mongoose model for users.

Responsibilities:

- defines user schema fields
- adds validation rules

### `src/models/report.model.ts`
Mongoose model for uploaded reports.

Responsibilities:

### `src/routes/user.routes.ts`

## Request Flow

A typical request flows like this:

1. A route is defined in `src/routes`
2. OAuth middleware passes through when auth is disabled, or verifies the bearer token when auth is enabled
3. Validation middleware checks request data
4. The controller receives the request
5. The controller calls the service
6. The service reads or writes data using the model
7. Helpers shape the output if needed
8. A shared response utility sends the final response
9. Errors go to the centralized error middleware

## Main Report APIs

- `POST /api/v1/reports/upload`
- `GET /api/v1/reports/:reportId`
- `GET /api/v1/reports/:reportId/categories`
- `GET /api/v1/reports/:reportId/phenotypes/:phenotypeId`
- `GET /api/v1/reports/:reportId/categories/:categoryName/phenotypes`

## Notes For Future Development

- Keep controllers thin
- Put business logic in services
- Put data transformation logic in helpers
- Keep request validation in `utils/*.validation.ts`
- Update `src/docs/openapi.ts` whenever a route changes
- Add new route files only when a module becomes large enough to deserve its own area
